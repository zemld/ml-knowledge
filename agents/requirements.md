# Что нужно для корректной работы агентов

> Агент ломается не из-за одной причины — он деградирует по цепочке: слабая модель неправильно планирует, плохое описание инструмента вызывает не тот API, переполненный контекст теряет историю, отсутствие стоп-условия зацикливает систему.

## 1. Достаточно мощная модель

Планирование, выбор инструментов и интерпретация наблюдений требуют хорошего рассуждения. Слабые модели:
- путаются с форматом вызова инструментов,
- теряют цель задачи через несколько витков,
- галлюцинируют наблюдения вместо реальных результатов.

Практическое правило: для агентских задач нужна модель, хорошо справляющаяся с instruction-following и function calling.

## 2. Чёткое описание инструментов

LLM выбирает инструмент, читая его описание в системном промпте. Размытое описание → неправильный выбор → ошибочное действие.

Хорошее описание инструмента содержит:
- **name** — краткое уникальное имя,
- **description** — когда и зачем использовать (не «что делает», а «когда применять»),
- **parameters** — типы, обязательность, примеры значений,
- **возвращаемый формат** — что ожидать в Observation.

Плохо: `search(query)` — «выполняет поиск».  
Хорошо: `web_search(query: str)` — «ищет актуальную информацию в интернете; используй когда нужны свежие данные, текущие события или факты после даты обучения модели».

## 3. Управление контекстным окном

Каждый виток цикла добавляет Thought + Action + Observation в контекст. Через десятки итераций контекст заполняется и начальные инструкции или ранние наблюдения обрезаются.

Стратегии управления:
- **Summarization** — периодически сжимать историю витков.
- **Долгосрочная память (RAG)** — выносить факты в векторную БД, извлекать по необходимости.
- **Ограничение глубины** — не накапливать всё подряд, оставлять только последние N витков.

## 4. Условие остановки

Без него агент уходит в бесконечный цикл, тратя токены и деньги.

Обязательно задать хотя бы одно:
- **Максимальное число итераций** — жёсткий лимит (например, 20 витков).
- **Финальный токен** — агент генерирует `Answer:` → система перехватывает и завершает цикл.
- **Таймаут** — принудительное завершение по времени.

## 5. Обработка ошибок инструментов

Инструменты падают: API возвращает 500, поиск не даёт результатов, база недоступна. Агент должен получить информативный Observation (`"Error: API rate limit exceeded"`) и суметь переключиться на альтернативное действие.

Без явного формата ошибки модель может интерпретировать сбой как успешный результат и двигаться дальше с неверными данными.

## Типичные симптомы поломки

| Симптом | Вероятная причина |
|---|---|
| Вызывает не тот инструмент | Плохое описание tools |
| Бесконечный цикл | Нет условия остановки |
| Теряет цель через 5+ витков | Слабая модель / переполнен контекст |
| Галлюцинирует Observation | Модель не ждёт реального ответа инструмента |
| Один и тот же Action повторяется | Нет проверки на дублирование шагов |
