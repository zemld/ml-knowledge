# RAG — Retrieval-Augmented Generation

> RAG — архитектурный паттерн, при котором перед генерацией ответа LLM выполняет поиск по внешней базе знаний и добавляет найденные документы в промпт, снижая галлюцинации и устраняя ограничение дату обучения.

## Зачем нужен RAG

У LLM есть два фундаментальных ограничения:

1. **Устаревшие знания** — модель знает только то, что было в обучающих данных до даты среза. Новые факты, внутренние документы компании, актуальные данные — недоступны.
2. **Галлюцинации** — когда модель не знает ответа, она может уверенно генерировать правдоподобный, но ложный текст.

RAG решает оба: вместо того чтобы полагаться на «параметрическую память» (веса модели), он добавляет **«непараметрическую память»** — внешний индекс, который можно обновлять независимо от модели.

## Две фазы архитектуры

RAG разделён на два независимых пайплайна:

```
ОФЛАЙН (Ingestion)              ОНЛАЙН (Query)
────────────────────            ──────────────────────────────
Документы                       Запрос пользователя
    ↓                               ↓
Разбивка на чанки               Эмбеддинг запроса
    ↓                               ↓
Эмбеддинг чанков                Поиск по векторному хранилищу
    ↓                               ↓
Сохранение в                    Топ-K чанков
векторное хранилище                 ↓
                                Дополненный промпт
                                    ↓
                                LLM → Ответ
```

## Компоненты системы

| Компонент | Роль |
|---|---|
| **Документы** | источник знаний (PDF, HTML, Markdown, БД …) |
| **Chunker** | разбивает документы на фрагменты управляемого размера |
| **Embedding-модель** | преобразует текст в плотный вектор смыслов |
| **Векторное хранилище** | индексирует и хранит эмбеддинги для быстрого поиска |
| **Ретривер** | находит Top-K наиболее похожих чанков по запросу |
| **Augmentation** | вставляет найденные чанки в промпт к LLM |
| **LLM-генератор** | отвечает на вопрос, опираясь на предоставленный контекст |
