# Компоненты RAG

> Каждый блок RAG-системы отвечает за один шаг преобразования — от сырых документов до финального ответа пользователю.

## 1. Документы (Knowledge Source)

Сырые источники знаний: PDF, HTML, Markdown, базы данных, таблицы, внутренние вики. Качество и актуальность документов определяют потолок качества всей системы.

## 2. Chunker (разбивка на чанки)

LLM не может обработать целую книгу как контекст, а embedding целого документа теряет конкретику. Chunker делит документы на **чанки** — небольшие фрагменты (обычно 256–1024 токена с перекрытием 10–20%).

Стратегии разбивки:
- **Fixed-size** — по числу токенов, просто и быстро.
- **Semantic** — границы там, где меняется тема; лучше для нарративных текстов.
- **Hierarchical** — документ → секции → абзацы; в 3–5× лучше для структурированных документов.

Перекрытие между чанками необходимо, чтобы ключевые предложения на границах не терялись.

## 3. Embedding-модель

Преобразует текстовый чанк в **плотный вектор** (dense vector) фиксированной размерности — числовое представление семантического смысла. Чанки с похожим смыслом получают близкие векторы.

Важно: **та же модель** должна использоваться для эмбеддинга документов при индексации и для эмбеддинга запроса при поиске — иначе векторы несовместимы.

Популярные модели: `text-embedding-3-small` (OpenAI), `BGE`, `GTE-Qwen2`.

## 4. Векторное хранилище (Vector Store)

База данных, хранящая пары `(вектор, метаданные + текст чанка)` и поддерживающая быстрый поиск ближайших соседей (ANN — Approximate Nearest Neighbor).

Внутри используются индексы типа **HNSW** или **IVF**, которые позволяют искать среди миллионов векторов за единицы миллисекунд.

Популярные решения: Qdrant, Pinecone, Weaviate, ChromaDB, OpenSearch.

## 5. Ретривер

Принимает эмбеддинг запроса и возвращает **Top-K** наиболее релевантных чанков. Сходство между векторами измеряется через косинусное расстояние или скалярное произведение.

Может быть плотным (dense), разреженным (sparse) или гибридным. Подробнее — в `[[rag/retrieval|Типах ретриверов]]`.

## 6. Augmentation (дополнение промпта)

Найденные чанки вставляются в промпт к LLM как **контекст**. Типичная структура промпта:

```
System: Ты — ассистент. Отвечай только на основе предоставленного контекста.

Контекст:
[Чанк 1]
[Чанк 2]
...

Вопрос: <запрос пользователя>
```

Качество augmentation зависит от того, насколько релевантны чанки — мусорный контекст сбивает LLM с толку.

## 7. LLM-генератор

Получает дополненный промпт и генерирует ответ, основываясь на предоставленном контексте. Модель не «придумывает» — она синтезирует информацию из чанков, что снижает галлюцинации.

LLM читает контекст через механизм cross-attention. Подробнее о self-attention и cross-attention — в `[[neural-networks/transformer/decoder|Декодере Transformer]]`.
