# Beam Search (лучевой поиск)

> Алгоритм декодирования последовательностей, который на каждом шаге удерживает $k$ лучших гипотез вместо одной, компенсируя близорукость жадного поиска.

## Проблема жадного декодирования

Жадный поиск (greedy search) на каждом шаге выбирает токен с наибольшей вероятностью:

$$\hat{y}_t = \arg\max_{y} P(y \mid y_{1:t-1}, x)$$

Это быстро, но локально оптимальный выбор часто ведёт к глобально плохому результату: однажды выбранный токен нельзя отменить, и модель может «застрять» в повторяющейся петле или упустить более вероятную последовательность в целом.

## Алгоритм

Beam search хранит $k$ гипотез (лучей) одновременно. На каждом шаге:

1. Каждый из $k$ лучей раскрывается: для него рассматриваются все возможные следующие токены.
2. Все $k \times |V|$ продолжений (где $|V|$ — размер словаря) оцениваются накопленной суммой лог-вероятностей:

$$\text{score}(y_{1:t}) = \sum_{i=1}^{t} \log P(y_i \mid y_{1:i-1}, x)$$

3. Оставляются только $k$ продолжений с наибольшим score — остальные отбрасываются.
4. Поиск останавливается, когда луч достигает токена конца последовательности `<EOS>` или максимальной длины.

По окончании из всех завершённых гипотез берётся та, у которой score наибольший.

Логарифмы используются вместо вероятностей, чтобы избежать численного underflow при умножении многих малых чисел.

## Параметр `num_beams`

`num_beams` ($k$) — ширина луча — главный управляющий параметр:

| `num_beams` | Поведение |
|---|---|
| 1 | Эквивалентен жадному поиску |
| 4–6 | Стандартный диапазон для ASR и перевода |
| >10 | Резкий рост памяти и латентности, качество перестаёт расти |

Память растёт пропорционально $k$: при больших моделях beam search с $k=4$ потребляет ~в 4 раза больше KV-кэша, чем жадный поиск.

**Beam search curse:** улучшение резкое при переходе от $k=1$ к $k\approx4$, после чего качество выходит на плато и может падать — широкие лучи начинают исследовать области вне обучающего распределения.

## Штраф за длину (Length Penalty)

Без коррекции beam search предпочитает короткие последовательности: каждый новый токен прибавляет отрицательное слагаемое к score. Для нейтрализации вводят нормировку:

$$\text{score\_norm}(y_{1:T}) = \frac{\sum_{t=1}^{T} \log P(y_t \mid \ldots)}{T^\alpha}$$

где $\alpha \in [0.6, 1.0]$ — гиперпараметр. При $\alpha = 0$ нормировки нет (короткие побеждают), при $\alpha = 1$ — полная нормировка по длине.

## Ограничения

**Повторения.** Beam search склонен к повторяющимся фразам: высоковероятные продолжения часто образуют петлю. Лечится $n$-gram блокировкой (запрет повторять уже встретившийся $n$-gram).

**Однообразие.** Несколько лучей часто сходятся к похожим фразам — разнообразия нет. Для творческих задач лучше подходит сэмплинг с температурой.

**Вычислительная стоимость.** Растёт линейно с $k$. Производственные чат-системы (OpenAI, Anthropic) обычно не используют beam search, предпочитая один луч с сэмплингом — это сохраняет пропускную способность.

## Когда использовать

| Задача | Рекомендация |
|---|---|
| ASR, машинный перевод, суммаризация | Beam search ($k=4$–$8$) |
| Генерация кода | Beam search или жадный |
| Чат, творческая генерация | Сэмплинг с температурой / nucleus sampling |
